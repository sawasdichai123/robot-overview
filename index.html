* Why ASK Reads 0% — Not a Bug
C2 shares C1's prompt, LLM options, and fallback verbatim (same two Jul 1 pipeline bugs were found still present here and fixed identically). The only difference is override_ask_to_execute(), which intercepts every ASK decision and immediately executes on the first candidate object — before it is logged. C2 can therefore never match an ASK ground-truth label by definition; this is the defining property of the "no-clarification" baseline, not a regression.
Object-Identity Bias Discovered
Task success on the standard scene (18.2%) was far below the ~50% expected from a random guess. Detection logs showed the water bottle was listed first — and therefore always picked — in nearly every trial, regardless of which side it was on. Re-running the ASK-only batches with the bottles physically swapped confirmed this: the water bottle was still picked first, and task success flipped to 78.6%. The bias tracks the object (HSV detects the water bottle's lighter color more reliably), not the table side — averaging both arrangements (46.4%) recovers the expected near-chance baseline once the bias is controlled for.

Local ≤7B Matches Cloud-Tier Accuracy
Same prompt, same 30-case set, same physical scene as C1 — only the decision backend changed (local qwen2.5:7b → cloud Claude Sonnet 5 via the Anthropic API). Decision accuracy is identical (100%) on both; the cloud model is faster per-call (4.3 s vs 11.8 s avg) but requires internet access and per-token billing, which the local system avoids entirely. This is the core "local is sufficient" comparison for the paper.
Cloud Backend Went Through 3 Providers Before This Run
GPT-4o (GitHub Models) → Gemini → GPT-4o → Claude Sonnet 5 in a single day. Gemini's free tier capped at 20 requests/day (too small for 30 trials); GitHub Models' free tier (50/day) got exhausted by same-day testing and silently fell back to rule-based decisions mid-experiment (that data was discarded as invalid); personal GitHub accounts cannot pay for more quota (Enterprise-only). Claude Sonnet 5 was chosen as the closest price/capability match to GPT-4o for a fair comparison, and supports normal pay-as-you-go billing on a personal account.

Vision Grounding Is Necessary (H3 Confirmed)
Same model, same prompt structure, same 30-case set as C1 — the only change is that ai_processing_node_c4.py never receives the YOLO+HSV object list, and the ROS node doesn't even subscribe to /detected_objects (so vision can't leak back in through the post-decision guard). Decision accuracy drops to 80.0%, and every single error is an over-ask — the model never confidently executes or rejects the wrong thing, it just can't rule out ambiguity or absence that vision would resolve instantly. The cleanest evidence: both REJECT misses are empty-table scenes — knowing no object is present is definitionally impossible from speech alone. Ambiguity detection from language stays perfect (11/11 ASK), but recovery breaks down — task success on those ASK commands falls to 36.4%, since a confirmed position can't be checked against reality without vision (the arm's own IK still aims at a real marker when it does execute — only the decision layer is blind, not the whole robot).
Confound Check: Ruled Out Prompt Wording as the Cause
The four EXECUTE misses all state a position clearly ("หยิบขวดขวา," "Grab the bottle on the left") — the system prompt's own rules say a stated position never needs vision to resolve. Before trusting the 60% figure, a real confound was checked: the ablation's disclaimer text was more heavily hedged than C1's own reminder sentence, raising the chance that wording, not missing vision, drove the extra caution. The disclaimer was rewritten to a minimal, C1-parallel form, and a quick isolated re-run of just the 4 failed cases gave a partial, inconclusive flip (2 of 4 changed). To resolve the ambiguity, all 30 cases were re-run in full under the revised prompt — the result reproduced exactly: 80.0%, the identical six failing cases, including the two that had flipped in isolation. This confirms the isolated flip was ordinary LLM run-to-run variance, not a wording effect, and that 80.0% is attributable to the missing vision, not to how its absence is phrased.

Implication for the Paper
The offline benchmark (Section V-C) used the same C24 "อะไรก็ได้" case as REJECT ground truth; the live experiment (Jul 1) revised that label to ASK — the system can meaningfully ask "what would you like me to pick up?" instead of refusing outright. Once the ground truth was corrected and two pipeline bugs were fixed (a truncated-JSON fallback and a stale blue-only rule incompatible with real, non-color-coded test objects), the live system reached 100% across all three decision categories, confirmed consistent across 3 repeated runs. End-to-end decision latency (avg 11.8 s) is dominated by LLM inference (≈9–10 s on Mac M3 via Ollama) plus ROS pub/sub overhead — within acceptable range for lab pick-and-place tasks.
H1 Confirmed — Clarification Is Necessary
C2 (identical pipeline, minus clarification) matches C1 exactly on EXECUTE/REJECT — confirming the gap is caused by the missing clarification step, not by any other difference. On ambiguous commands, C2 doesn't fail randomly: its "first-match" guess is systematically biased toward whichever object the vision pipeline detects more confidently (18.2% vs. 78.6% success depending on scene arrangement, averaging 46.4%). A deployed no-clarification system would therefore misinterpret ambiguous commands in a predictable, user-intent-independent way — a stronger argument for asking than "clarification is merely better than a coin flip."
H2 Confirmed — Local ≤7B Matches Cloud-Tier Accuracy
C1 and C3 (Claude Sonnet 5, byte-identical prompt) both reach 100% (30/30) — the local model gives up nothing in decision quality against a substantially larger cloud model. The only real difference is latency (11.8s local vs. 4.3s cloud), a hardware/inference-engine gap, not a reasoning-capability gap — and one that is offline-vs-online by nature, since C3 cannot function at all without an internet connection.
H3 Confirmed — Vision Grounding Is Necessary
C4 (no vision) drops to 80.0% vs. C1's 100%, and every error is an over-ask — the model never confidently executes or rejects the wrong thing. The cleanest evidence: both REJECT misses are empty-table scenes, a fact that is definitionally unrecoverable from speech alone. A suspected prompt-wording confound was explicitly checked and ruled out — a full 30-case re-run under a revised, less-hedged prompt reproduced the identical 80.0% and the same six failing cases.
Pipeline Optimization (Jul 3, 2026) — Latency Figures Above Predate This
After C1/C2 data collection, the harness and LLM-serving stack were tuned: skipping a redundant 30s arm-wait on REJECT trials, tightening the LLM timeout (120s → 60s+25s retry) with an echo-based guard against stale decisions crossing trial boundaries, and enabling flash-attention + 8-bit KV-cache quantization on the Ollama server. The last change alone measured ~35% faster token generation (9.3 → 12.6 tok/s, like-for-like benchmark). None of these touch decision logic, the system prompt, or parsing — a 5-case live smoke test across EXECUTE/ASK/REJECT confirmed no accuracy regression. The 11.8s (C1) / 11.2s (C2) average latencies shown above were measured before this change, so future conditions (C3/C4) collected under the faster configuration will not be directly latency-comparable without accounting for it — decision-accuracy comparisons are unaffected. See docs/pipeline_optimization_plan.md for full detail.
Phase 2 Optimized Variant (Jul 7, 2026) — Validated, Not a Replacement for C1's Headline Number
After all four conditions (C1–C4) were collected, tts_response was removed from the JSON schema the LLM must produce (it was always discarded and rebuilt from a template regardless of the model's output — see the Discussion's latency-bottleneck finding), num_predict cut 150→80 accordingly, and Ollama's format:'json' enabled to guarantee valid output. All 4 required validation steps passed: SYSTEM_PROMPT parity between C1/C2 re-confirmed identical, the 30-case offline benchmark scored 100.0% (up from 96.7%), and a full live 30-trial C1 re-run scored 100.0% (30/30) — identical to the original C1 result — at 9.43 s mean / 9.05 s median decision latency, a 20.1% reduction from the original 11.8 s. Reported here as a validated efficiency improvement only: C2/C3/C4 were not re-collected under this schema, since the change is decision-logic-neutral and doesn't affect any of the paper's accuracy-based conclusions (H1–H3).
